BattleGroup 数据流和权威边界

这张图是核心。长期状态进战斗前被冻结成快照；Runtime 不知道存档对象；结算从事件和结果回写。

长期状态Hero、Corps、BattleGroup、City、Equipment 是存档权威。
运行时状态HP、坐标、目标、命令、AI、冷却只存在于 Runtime。
结算边界只有完整 Outcome + 一致 EventStream 才能进入 Settlement。

Ability / Effect 资源化管线

技能、消耗、冷却、目标、效果、标签、Modifier 是定义。Runtime 只持有执行态；结算和战报共享事件事实。

Machinations 风格资源与成长闭环

城市资源、训练、装备等级、战损、恢复和奖励回写必须能画成闭环，不能各系统独立造账。

战报归因：同一事件源

战报和结算不能各算各的。Runtime 的 EventStream 是共同事实源，Settlement 和 Report 各自消费。

迁移边界：旧实现只能适配，不能污染核心合同

旧 Force/Garrison/AutoBattle 可以作为迁移输入，但不能成为新 Domain 或 Runtime 的基础抽象。

核心规则

  • 新架构不迁就旧概念。
  • 旧数据只能转换成新快照或新状态。
  • 同一职责不能保留双权威。

失败规则

  • 映射缺失必须显式失败。
  • 不能用隐藏 fallback 维持表面可运行。
  • Adapter 必须可整体移除。
