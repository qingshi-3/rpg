# Engineering Tech Debt Register

This document tracks engineering debt: gaps between the current implementation and durable engineering standards such as cohesion, ownership clarity, resource authoring discipline, explicit failure, and regression guard coverage.

This register is intentionally separate from `gameplay-alignment/gap-register.md`. `gap-register.md` tracks implementation or document drift against the accepted gameplay direction; this file tracks implementation quality debt that can exist even when gameplay direction is correct. The two registers do not form dual authority. Gameplay and system authority still come from `gameplay-design/`, `system-design/`, and the proposal flow defined by `AGENTS.md`.

For code work, "pure engineering refactor" below means no design proposal is required if behavior and accepted architecture stay unchanged. The repository still expects a focused implementation proposal before code, scene, resource, or test changes begin. Rows marked as architecture-boundary work need the design proposal flow before implementation if the fix changes ownership contracts, scene/resource taxonomy, or runtime authority.

## Scoped Audit Notes

- Runtime hotspot A is a real cohesion problem. `BattleRuntimeTickResolver` is split across multiple partial files and is about 2k lines in the current tree. It owns tick orchestration plus target acquisition, AI request shaping, local combat perception, engagement transitions, movement proposal/commit, attack application, plan-state events, retargeting, diagnostics, and event construction.
- Presentation hotspot B has multiple large root-like nodes. `WorldSiteRoot*.cs` is about 6.8k lines across many partial files; `BattleUnitRoot`, `BattleGridHighlightOverlay`, and `UnitAnimationComponent` are each near 800 lines locally.
- In the scoped Presentation files, this audit did **not** find direct construction of a full `Control` tree via `new Control`, `new Button`, `new VBoxContainer`, etc. `WorldSiteRoot.BuildSiteHud()` instantiates `WorldSitePeacetimeHud.tscn`, and repeated buttons/labels are mostly instantiated through `GameUiSceneFactory` packed-scene helpers.
- Remaining concrete resource-authoring pressure points are root-owned dynamic row binding and procedural non-`Control` drawing: `WorldSiteRoot.BattlePreparationHud.cs`, `WorldSiteRoot.BattleRuntimeCommandHud.cs`, and `WorldSiteRoot.SiteManagementHud.cs` create repeated button rows in the root partial; `BattleGridHighlightOverlay.cs` builds runtime `Node2D`, `Polygon2D`, and `Line2D` nodes directly for vector overlays. These are not the exact forbidden "hardcoded Control tree" case, but they should be guarded and isolated before they grow further.

## Suggested Repair Order

1. TD-005 first. It is the cheapest, behavior-preserving guard and directly protects the AGENTS.md resource-authoring rule before larger Presentation refactors move code around.
2. TD-003 next for Runtime. Attack/movement/event extraction has strong existing behavioral tests and creates concrete seams without changing target policy.
3. TD-002 after TD-003. Target selection, perception, and engagement are more policy-sensitive; do them after the tick resolver has phase-service seams and performance counters still prove behavior stays bounded.
4. Close TD-001 as the umbrella only after TD-002 and TD-003. The resolver-size guard should be added when the extracted boundaries are real, not as a cosmetic line-count cleanup.
5. TD-004 after the UI guard, preferably one collaborator at a time: command HUD binder, battle-preparation presenter, objective-planning presenter, deployment drag controller, runtime playback presenter, then site-management binder. Stop and use the design proposal flow if any step changes host ownership or scene/resource taxonomy.
6. TD-006, TD-007, and TD-008 can proceed independently after TD-005. Do TD-006 before deep animation/overlay work if live battle presentation is the active risk; do TD-007 first if hover/highlight behavior is changing; do TD-008 first only when animation authoring or fallback behavior is the current bottleneck.
