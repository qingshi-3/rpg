# rpg

使用 Godot 4.5 .NET 与 C# 开发的 2D 像素风多种族幻想战略经营游戏。

> 玩家作为君主，在实时运转的大世界中经营领地、组织英雄军团、争夺战略地点，并进入城域实景地图进行建设与低频实时战术作战。

本 README 只提供产品与项目入口，不承担玩法或架构权威。发生冲突时，以 [权威路由](gameplay-alignment/authority-map.md) 指向的文档为准。

## 游戏定位

游戏由三个相互闭环的核心支柱组成：

| 核心支柱 | 内容比重 | 玩家关注点 |
|---|---:|---|
| 英雄、人物与军团 | 40% | 英雄身份、军团搭配、成长、装备与长期战损 |
| 城池和战略地点经营 | 30% | 领土、城市定位、资源、建设、路线安全与势力扩张 |
| 战斗编组与实时指挥 | 30% | 战前编组部署、地形利用、中频命令、战术暂停与英雄技能 |

它不是纯城建、纯英雄 RPG、传统回合制战略、纯自动战斗，也不是依赖高频操作和单兵微操的重型 RTS。玩家决定战略目标和战术意图，系统负责士兵层面的移动、接敌、占位与局部支援。

玩家自身代表君主和势力意志，不作为战斗单位参战。

## 核心循环

```text
观察大世界局势
-> 管理城池、英雄、军团和资源
-> 选择目标并派遣远征军
-> 在大世界行军、侦察、拦截或遭遇事件
-> 进入目标实景地图，完成部署和实时战斗
-> 通过战报理解结果，并将战果写回领土、城池、资源和军团状态
-> 恢复、建设、招募和成长
-> 再次扩张
```

## 大世界与战略地点

大世界是一张实时运转的战略沙盘。军队、敌方行动、资源结算和事件随大世界时间继续推进；进入城池管理、战斗准备、正式战斗、对话或报告等聚焦状态时，大世界时间暂停，返回后继续运转。

玩家在大世界中关注：

- 城市、关隘、资源点、遗迹、地下城、机会事件及势力边界；
- 军队行军、敌军进攻、野外拦截、视野与战争迷雾；
- 道路、河流、山口等交通条件和战略阻隔；
- 资源控制、远征方向、领地扩张和地点间的支援关系。

长期方向上，城市不只是地图图标，而是包含主城区、周边区域、哨站与防区、资源区及对外通道的完整城域。外围控制设施应影响视野、路线和提前拦截；外围失守则可能让敌军更接近主城区，并影响实景战斗的入口和部署态势。

只有城市承担完整经营。关隘、资源点、遗迹、地下城和机会事件使用更轻量的控制、驻军、奖励、挑战或限时规则。

## 城域经营

城市背后对应一张更精细的实景地图。这张地图长期统一承载：

- 建筑建设和城市发展；
- 农田、矿区、林场等资源开发；
- 驻军、哨站、城防和部署区域；
- 敌人进攻时的攻城与防守战斗。

经营感主要来自领土控制、城市定位、资源禀赋、路线安全、英雄与军团配置，以及战争造成的长期后果，而不是复杂的工人搬运或 RTS 式手动采集。

城市可以发展成主城、边境要塞、资源城、工坊城或特殊兵种城市。建设区域、建筑位置、城市能力和当地地形应共同影响经营与防守，但不会演变成完整的工人生产链模拟。

## 英雄、军团与战斗组

正常战斗编组为：

```text
1 个英雄 + 1 支主军团 = 1 个战斗组
```

玩家控制战斗组、英雄和军团的命令层级，不逐个操作士兵。目标指挥结构分为：

- 英雄命令：移动、攻击、坚守、撤退和施放技能；
- 军团命令：推进、守卫、占领区域、保护英雄和撤退；
- 整体命令：战斗组移动、进攻、防守、撤退和重整。

军团是持久化的战略对象，保留兵力、等级、装备等级、经验、恢复状态、驻地、英雄归属和远征状态。战斗中的伤亡、溃退与恢复需求会回到战略层，而不是在离开战场后自动清零。

## 实时战术战斗

战斗是在方格空间上持续运行的低频实时战术战斗。玩家负责中频率的指挥决策：

- 战前选择参战战斗组、阵型、部署位置和初始目的地；
- 战斗中选择或多选战斗组，调整目的地和局部战线；
- 使用英雄技能，把握冷却、资源和关键释放时机；
- 利用桥梁、城门、山口、森林、高地和多入口地形；
- 决定何时投入或保留力量、转移战线、撤退或重整；
- 使用战术暂停处理关键命令。

士兵在命令范围内自动寻路、移动、接敌、争夺合法位置并响应局部战斗。玩家决定要去哪里、守住哪里、何时改变战线和使用关键能力，系统负责具体执行。因此它是“英雄带兵的轻 RTS”，不是回合制战棋，也不是部署后只能观看的自动战斗。

战斗结果来自运行中的单位、命令、伤害、技能和目标事实。基础战报记录结果、战损、技能使用和失败候选原因，再由结算流程把地点控制、奖励、军团状态及资源变化写回战略层。

## 成长与扩张

玩家通过以下方式发展势力：

- 占领城市、资源点、关隘和特殊地点；
- 建设训练场、市场、农场、矿场、箭塔和医疗设施等建筑；
- 招募英雄，创建、补充和调整英雄的主军团；
- 控制特殊资源地，并在城市建设对应设施以解锁特殊军团路线；
- 提升英雄等级、阶位、技能和装备；
- 提升军团训练等级、装备水平、经验和战后恢复能力；
- 探索遗迹和地下城，获得装备、材料、兵种来源和剧情内容。

兵种的战场职责与幻想形态分离：步兵、盾兵、弓兵、骑兵、法师等职责可以承载龙骑、机械、亡灵、野兽或其他种族形态。英雄、军团、技能、装备、建筑和地点内容应通过定义与配置扩展，而不是依赖一次性硬编码事件。

## 核心辨识度

大世界上的城域、资源、道路和进攻方向，应映射到实景地图中的地形、建筑、入口、部署和战斗。经营与战争使用同一套地理事实：玩家在战略层争夺的路线、设施与区域，会在进入实景地图后继续产生可见影响，而不是变成两个互不相关的系统。

## 当前实现基础

当前代码已经形成以下可继续扩展的玩法闭环：

- 实时大世界时钟、军队行军、目标到达、野外军队拦截、视野与战争迷雾；
- 城市控制、共享资源、建设区域、建筑放置与产出、兵力容量和被动预备兵恢复；
- 英雄与持久军团、招募或更换主军团、军团补充、远征创建和目标调整；
- 战略远征到战斗准备、战斗运行、结算报告和战略写回的正式交接链路；
- 战前编组部署、阵型占位、初始目的地，以及战斗中的多选目的地命令；
- 方格实时战斗、单位体型与占位、编译导航、局部移动、攻击、英雄技能和战术暂停；
- 基于战斗事件的基础战报，以及地点控制、奖励、军团战损和反馈写回；
- 战略状态的版本化保存与读取基础能力。

当前玩家指挥能力已经覆盖部署、目的地、多选、战术暂停和英雄技能，但完整的英雄命令、军团命令与整体命令词汇仍在继续扩展，不能视为全部完成。

## 仍在继续设计

- 城域内部的主城区、外围区域、哨站、防区和资源区控制关系；
- 哨站战败、提前拦截、撤退和敌军直逼主城区时的规则；
- 跨城市资源运输、道路安全、区域物流和补给深度；
- 大世界地理、城域边界、道路与实景地图地形、入口、建筑和部署区的精确映射；
- 完整的英雄/军团分离指挥、预备力量投入和更多战术姿态；
- 多种族内容、英雄关系、装备成长、特殊兵种来源、遗迹、地下城与剧情的内容规模。

## 本地运行

1. 使用 Godot 4.5 .NET 打开仓库根目录下的 `project.godot`。
2. 运行项目主场景 `scenes/world/StrategicWorldRoot.tscn`。
3. 如需单独检查进入地点后的管理、部署或战斗流程，可运行 `scenes/world/sites/WorldSiteRoot.tscn`。

使用 Rider、Visual Studio 或其他 C# IDE 时，优先打开 `rpg.sln`。命令行构建使用低并发：

```powershell
dotnet restore rpg.sln
dotnet build rpg.sln -maxcpucount:2 -v:minimal
```
