# 核心玩法方向

本文记录当前稳定产品方向。系统细节放在 `../architecture/`、`../world/`、`../battle/` 和 `../character/`，不要把阶段性方案和内容清单堆进本文。

## 核心定位

项目定位为：

```text
《异世界的召唤》。
召唤群像关系驱动的战略 RPG。
底层世界按势力、场域、部队、召唤者、关系和事件生态自动运转。
玩家是有限介入者，不是全局操作员。
重要 WorldSite 是可进入、可战斗、可经营、可被世界状态改变的小型运行空间。
```

三群式系统在本项目中的价值是大世界生态骨架：`Faction`、`WorldSite`、`WorldArmy`、威胁、攻防、结盟、遭遇、战斗入口和结果回写。它不是题材目标，也不是要求玩家默认扮演君主或完成统一天下。

召唤机制是产品核心差异。外来人物可以来自三国、水浒、幕末或原创世界，但来源只提供展示、内容分组和叙事语境。运行时系统只读取抽象关系、社交属性、位阶、职责、天分和控制权，不按具体人名写特殊逻辑。

## 玩家身份

玩家身份可以由世界观和开局背景定义，不应被提前锁死成君主、魔王或单纯冒险者。

稳定的系统身份是：

```text
有限介入者 / 召唤锚点 / 组织核心
```

玩家能影响召唤、任命、亲自出战、世界事件和关系走向，但不能也不应该直接优化所有角色、所有战斗和所有场域。

玩家主要问题不是：

```text
我该派哪支军队攻哪里？
我该把所有角色都养到最优吗？
```

而是：

```text
这个局势值不值得我亲自介入？
我让谁成为核心亲控单位，谁去驻守、带队或自治？
这个召唤者的旧关系在新世界里会变成资产、风险还是裂痕？
我放弃处理的事件会由谁自动处理，后果是什么？
```

随着成长，玩家可以获得更多召唤权限、任命权限、亲控席位、区域管理权和世界介入手段。扩大控制力必须伴随成本和限制，不能把全员精养变成最优策略。

## 世界结构

本项目不采用章节式线性结构。

设计应提供：

- 一个稳定的大世界背景和局势：族群仇恨、势力目标、召唤裂隙、资源压力、旧恩怨和本土矛盾。
- 一套世界运行逻辑：`WorldTick`、`WorldArmy`、`WorldSite`、威胁、机会、关系事件和条件触发大事件。
- 若干可能发生的重大事件：类似黄巾起义、董卓入京这类事件可以存在，但应按条件、时间窗或局势触发，而不是线性章节推进。

玩家自由操作应改变事件的触发时机、参与者、结果和后续世界状态。

## 核心循环

```text
观察大世界态势
-> 选择召唤、任命、行军、驻守、亲自介入或放权处理
-> 发现 WorldSite 状态、野外机会、召唤者关系、NPC 势力冲突或威胁
-> 选择介入方式
-> 进入场域、野外遭遇、交涉或战斗
-> 达成目标、撤退或失败
-> BattleResult / EventResult 回写世界、角色、资源、关系、位阶或威胁
-> 世界继续推进并产生下一轮局势
```

玩家成长的价值应体现在“更高层级的世界介入权”上，包括可召唤的对象、可任命的职责、可亲控的核心单位、可治理的区域、可进入的场域、可谈判的对象、可影响的战场、可掌握的资源和可承担的后果。

## 大世界生态

大世界负责让世界自己动起来：

- `WorldSite` 保存控制权、设施、资源、驻军、损毁、NPC / 对象记忆和威胁。
- `Faction` 保存敌友关系、目标、行动倾向和世界层归属。
- `WorldArmy` / `Expedition` 在大地图地表连续移动，承担出征、拦截、撤退、运输和追击。
- `ThreatPlan` 描述敌方计划，不直接等同于地图实体。
- `Opportunity` 描述短期机会，例如追杀、救援、秘境入口、资源成熟、会战侧翼或战后残场。

大世界不是静态任务板，也不是完整 4X。第一阶段只需要足够支撑玩家在运动中的世界里做有后果的选择。

## 场域原则

`WorldSite` 是本项目的重要差异化。

它既不是三群式薄节点，也不是只有经营按钮的据点 UI，而是可进入的小地图：

```text
大地图节点
-> 进入 WorldSite 小地图
-> 与 NPC、设施、资源点、入口和事件互动
-> 非战时经营、布防、修复、驻守或整备
-> 战时承载回合制局部战斗
-> 结果回写 WorldSiteState
-> 回到大地图继续推进
```

普通场域可以复用地图模板和通用交互点。关键场域才需要手工设计更完整的地形、NPC、设施、事件和战斗目标。不要把每个场域都做成大型关卡，也不要把场域退化成全屏经营菜单。

## 野外优势

本项目相比纯战略游戏的优势，是可以在场域之外做更丰富的野外交互。

场域之间的地表应逐步承载：

- 两支 `WorldArmy` 接触后的拦截或三方遭遇。
- 商队、修士、妖兽、流民、追杀队、巡逻队等移动对象。
- 短期机缘、伏击、救援、夺取、护送、侦察和撤离目标。
- 战后残场、临时营地、封锁线、山口和桥梁等空间机会。

野外内容优先建模为 `Opportunity` / `EncounterRequest`，只有需要长期保存运营状态的地点才升级为 `WorldSite`。

## 战斗定位

战斗是局部局势的结算方式之一。

战斗应继续保持：

- 回合制网格。
- AP 统一资源。
- 清晰的敌人意图。
- 地形、目标和机制驱动。
- 结构化 `BattleStartRequest` / `BattleResult`。
- 亲控单位数量有限。
- 非亲控友军按规则、Intent 或行为包自动行动，玩家只能通过宏观指令、资源或技能影响其计划。

但战斗目标不应长期只有“清空敌人”。更适合本方向的目标包括攻占、救援、护送、斩首、夺取、拖延、防守、撤离和破坏设施。

自动友军的核心要求不是黑箱智能，而是可预期、可预览、可解释。已有 battle Intent / preview 体系应扩展到非亲控友军，保证玩家能读懂计划、用有限资源修正计划并承担结果。

## 第一阶段方向

当前 `Strategic World V1` 仍然是正确的基础切片，但它的定位需要理解为“世界生态和场域回写骨架”，不是最终产品形态。

短期继续优先验证：

- 大地图地表和 `WorldArmy` 连续移动。
- 玩家小队 / 敌方部队接触产生遭遇。
- `WorldSiteRoot` 支持战时和非战时。
- 场域设施、驻军、损毁和单位部署能持久保存。
- 战斗结果能回写场域、部队、威胁和资源。
- 场域经营逐步落到小地图交互点，而不是只停留在右侧按钮。

## 明确不做

第一阶段不做：

- 纯三群式统一天下主玩法。
- 完整 4X。
- 完整开放世界模拟。
- 线性章节主线。
- 全员重养成和全员手操。
- 大量资源生产链。
- 大量手写线性剧情。
- 把场域做成只有菜单按钮的经营 UI。
- 为单个场域、单个角色或单个事件写一次性硬编码流程。
