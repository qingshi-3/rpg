# 机制战切片

本文定义下一阶段优先实现的战斗体验切片。目标不是做完整 Boss 框架，而是用最小可玩内容验证战斗核心是否成立。

## 目标

做一场 3-5 分钟可重复玩的机制战：

```text
矿洞亡灵督军 / 魂火矿柱
```

玩家不作为普通单位上场，而是阵营指挥者。场上英雄和部下负责站位、承伤、攻击；玩家使用有限的法术 / 卡牌 / 支援资源介入战局。

## 核心体验

每回合都要让玩家面对明确问题：

- 先打 Boss，还是处理魂火矿柱？
- 先救濒死友军，还是留资源打断蓄力？
- 先清小怪，还是封住危险区域？
- 根据敌人意图，这回合应该移动、控制、护盾还是爆发？

战斗成立的关键不是数值复杂，而是：

```text
读懂计划 -> 分配有限资源 -> 反制机制 -> 承担后果
```

## 战斗设定

地图：小型矿洞战场。

敌人：

- 1 个亡灵督军。
- 2-3 个普通亡灵。
- 可选 1 个祭司或矿柱守卫。

战场物件：

- 2-3 个魂火矿柱。

胜利目标：

- 击败亡灵督军。

失败压力：

- 友军全灭。
- 或 Boss 多次完成魂火爆发，导致战局不可逆。

## 核心机制

魂火矿柱拥有能量层数。

建议第一版规则：

- 每回合开始或 Boss 行动时，Boss 从一个矿柱吸取 1 层魂火。
- 矿柱或 Boss 能量达到 3 层时，Boss 下回合释放大范围魂火爆发。
- 爆发前必须通过 Intent 明确展示。
- 玩家可以用法术降低矿柱能量、打断蓄力、推开敌人或保护友军。

机制要点：

- Boss 大招必须可读。
- 玩家必须有至少两种反制方式。
- 反制方式之间要有资源冲突。
- 小怪要推动玩家分心，而不是只当血包。

## 玩家介入

第一版只做固定法术，不做完整卡组。

建议 4 张固定法术：

- 火矢：单体伤害；对蓄力目标有额外打断值。
- 护盾：给一个友军临时护盾。
- 震荡：推开敌人，或削减目标蓄力。
- 净化：降低一个魂火矿柱的能量。

资源建议：

- 每回合获得固定 2 点指挥 / 法力资源。
- 每张法术消耗 1-2 点。
- 不做抽牌、不做弃牌、不做牌库循环。

## 实现边界

优先使用已有扩展点：

- Effect
- Condition
- TargetRule
- Definition: Card, Ability, Rule

不要为第一版机制战修改：

- Battle flow
- AP system
- TurnSystem

如果发现必须修改这些核心流程，先写技术变更说明，再决定是否做。

## 可接受的第一版实现

第一版可以很朴素：

- 矿柱可以是战斗场景中的特殊实体或地图对象。
- 能量层数可以先用运行时状态保存。
- Intent 可以先用现有意图系统显示 Boss 下一步。
- 法术可以先接在现有行动 / 能力系统上。
- UI 只要求能看懂当前可用法术、目标和矿柱层数。

不要求：

- 华丽特效。
- 完整 Boss 阶段系统。
- 完整卡牌构筑。
- 完整战斗奖励。
- 大世界入口联动。

## 验收标准

这场战斗完成后，应能回答：

- 玩家是否每回合都有有意义的选择？
- Boss 机制是否不依赖说明书也能通过 Intent 看懂？
- 玩家是否能通过有限资源改变下一回合危险？
- 小怪是否制造压力，而不是只延长战斗时间？
- 失败是否能让玩家知道自己哪里没处理好？
- 胜利是否来自反制机制，而不是纯粹数值碾压？

如果这些问题答案是否定的，优先调整机制，不继续扩展大系统。

## 后续接入点

机制战验证成立后，再考虑接入：

- 大世界侦察：提前显示 Boss 吸能目标或爆发范围。
- 部下派遣：战前降低一个矿柱初始能量。
- 情感系统：高忠诚角色提高战斗支援概率。
- 招募系统：救下矿工或亡灵叛逃者后开放招募。
- 战后结果：矿洞安全、资源产出恢复、NPC 关系变化。
